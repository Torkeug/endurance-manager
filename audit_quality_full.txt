app/api/iracing/event/route.js:8 — HIGH — SECURITY — Auth check is fail-open when env var unset.
  `if (API_KEY && authHeader !== \`Bearer ${API_KEY}\`)` skips the check entirely if
  IRACING_BRIDGE_API_KEY is not set — any unauthenticated request is accepted.
  Fix: Invert guard to fail-closed: `if (!API_KEY || authHeader !== \`Bearer ${API_KEY}\`) return 401`.

app/api/iracing/lap/route.js:9 — HIGH — SECURITY — Identical fail-open auth bypass as above.
  Fix: Same inversion.

app/api/register-driver/route.js — LOW — SECURITY — auth_user_id from request body cannot be session-verified.
  The endpoint comment correctly notes the user has no active session yet (email confirmation pending),
  so auth.uid() is null and session-based verification is not possible. However, any unauthenticated
  caller can POST a fabricated UUID as auth_user_id to attach a driver record to an arbitrary auth
  user. The auth_user_id unique constraint prevents duplicate records, but not the initial misuse.
  Fix: Consider verifying auth_user_id against a real freshly-created Supabase auth user via
  supabaseServer.auth.admin.getUserById() — reject if the user was not created in the last few minutes
  or if email is already confirmed.

app/api/notify-admins/route.js — MEDIUM — SECURITY — No authentication check on notification endpoints.
  Any unauthenticated HTTP client can POST to this (and notify-driver-approved, notify-admins-approval,
  notify-stale-sync) to trigger email sends — a spam vector.
  Fix: Require a valid Supabase session (getSessionAndDriver()) or a shared secret before dispatching.
